# Incident Report

## Title

Suspicious PowerShell Execution and Possible Malware Activity on Support Endpoint

---

## Executive Summary

On 2026-05-20, a suspicious PowerShell execution alert was generated by Wazuh for the endpoint `WIN11-SUPPORT-04` in the Support department.

The investigation indicates that the user `NORDSECURE\support.johan` received and opened a macro-enabled Word document named `Faktura_45892.docm`. Shortly after the document was opened, `WINWORD.EXE` started `powershell.exe` with `ExecutionPolicy Bypass` and an encoded command.

The endpoint then performed DNS lookups for suspicious domains and established HTTPS connections to external IP addresses. Wazuh File Integrity Monitoring detected suspicious files created in `C:\Users\Public\`, and Windows Security Events showed that a scheduled task was created.

The endpoint later established an SMB session with the internal file server `FILE01` and accessed files in the Support share. Attempts to access HR and Finance shares were denied, which indicates that least privilege controls helped limit the impact.

The incident should be treated as a high-severity security incident due to the combination of suspicious execution, external communication, dropped files, persistence attempt, and internal file server activity.

## Initial Alert

The first major alert came from Wazuh.

```text
Time: 2026-05-20 09:14:33
Agent: WIN11-SUPPORT-04
Agent IP: 10.10.40.24
Rule Level: 10
Rule Description: Suspicious PowerShell execution detected
User: NORDSECURE\support.johan
Process: powershell.exe
Parent Process: WINWORD.EXE
CommandLine: powershell.exe -ExecutionPolicy Bypass -EncodedCommand ...
MITRE Tactic: Execution
MITRE Technique: T1059 – Command and Scripting Interpreter
```

This alert is suspicious because Microsoft Word normally should not start PowerShell. This behavior is commonly seen in phishing attacks where a malicious Office document triggers command execution.

## Impact Assessment

The affected endpoint showed clear signs of compromise, including suspicious process execution, external communication, file creation, and persistence activity.

The file server `FILE01` was also involved because the affected user account established an SMB session and accessed files in the Support share.

There is no confirmed evidence of data exfiltration in the available logs. However, the access to internal file shares and the large external HTTPS transfer should be investigated further.

Least privilege controls reduced the impact because access to HR and Finance shares was denied.

---

## Containment Actions

The following containment actions should be performed immediately:

```text
Isolate WIN11-SUPPORT-04 from the network
Temporarily disable NORDSECURE\support.johan
Block 185.44.12.90 at the firewall
Block 185.44.12.91 at the firewall
Block 91.203.18.77 at the firewall
Block suspicious domains at DNS / proxy level
Preserve Wazuh, Windows, DNS, firewall, IDS, and file server logs
Collect suspicious files for analysis
Search the environment for the same IoCs
```

---

## Actions to Avoid Before Evidence Collection

The following actions should not be performed immediately:

```text
Do not delete suspicious files before collecting evidence
Do not reboot the endpoint before checking for volatile evidence
Do not wipe or rebuild the system before preserving logs and files
```

These actions could destroy forensic evidence and make the investigation harder.

## Final Assessment

This incident should be classified as a high-severity security incident.

The strongest evidence is the chain of events showing a macro-enabled Word document leading to PowerShell execution, external communication, suspicious file creation, persistence, and later SMB activity toward the internal file server.

The investigation also shows why multiple log sources are important. Wazuh provided the first alert, but DNS logs, firewall logs, Windows events, IDS alerts, FIM events, and file server logs were needed to understand the full attack chain.

The main conclusion is that the organization should treat this as a confirmed compromise of `WIN11-SUPPORT-04`, preserve evidence, contain the endpoint, block IoCs, and perform threat hunting across the rest of the environment.
